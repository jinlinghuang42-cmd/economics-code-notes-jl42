## Basic project logic

1. A Stata project should separate code, raw data, generated data, logs, tables, and figures.
The basic idea is that raw data should be kept separate from files generated by code.
2. The analysis should be reproducible from scripts rather than from manual changes.

In practice, data is never edited by hand. Do not manually change data in Excel or the Stata data
editor. Every transformation should written as a command in a do-file. The outputs should generated by
code as well, with tables and figures exported directly to the `output/` folder rather than
copied or pasted. **The analysis should split into separate do-files**, each handling one stage of the
work like: cleaning, analysis, exporting tables. **Then a `master.do` runs them in order**. This keeps the data flow
one-directional, where code is the only thing that moves data, and no step depends on manual intervention.

## Folder structure

A clear folder structure makes the project easier to understand and maintain.

A possible structure:

* `00_code/`: do-files and scripts.
* `01_raw/`: original data files.
* `02_intermediate/`: cleaned versions of individual data sources.
* `03_data/`: final analysis data.
* `04_tables/`: exported tables.
* `05_figures/`: exported figures.
* `06_logs/`: Stata log files.
* `07_docs/`: data notes, variable dictionaries, or source documentation.

The exact folder names can change across projects, but the logic should remain clear.

## Path settings

Paths should be defined at the beginning of the do-file.

Example:

```stata
global root "path/to/project"

global code "$root/00_code"
global raw  "$root/01_raw"
global temp "$root/02_intermediate"
global data "$root/03_data"
global tab  "$root/04_tables"
global fig  "$root/05_figures"
global log  "$root/06_logs"
```

The goal is to avoid repeating long absolute paths throughout the script.

Example:

```stata
use "$raw/source_data.dta", clear
save "$data/analysis_data.dta", replace
```

## Log files

A log file records the commands and results produced by Stata.

Basic structure:

```stata
capture log close
log using "$log/main.log", replace text

* code here

log close
```

## Basic do-file structure

A do-file can be organized into repeated sections.

Possible order:

```text
preamble
→ path settings
→ log
→ data loading
→ data checks
→ data cleaning or analysis
→ exported outputs
→ log close
```

A simple skeleton:

```stata
****************************************************
* Project:
* File:
* Author:
* Date:
* Purpose:
****************************************************

version 18
clear all
set more off
capture log close

* Define paths
global root "path/to/project"

* Start log
log using "$log/main.log", replace text

* Load data
use "$data/analysis_data.dta", clear

* Data checks
describe
summarize

* Main code here

* Export outputs here

* Close log
log close
```

## Master do-file

For larger projects, a master do-file can be used to run other do-files in order.

Example:

```stata
do "$code/01_clean_sources.do"
do "$code/02_merge_sources.do"
do "$code/03_construct_variables.do"
do "$code/04_regressions.do"
```

The master do-file is mainly used to manage the workflow. It should make the whole project easier to rerun.
