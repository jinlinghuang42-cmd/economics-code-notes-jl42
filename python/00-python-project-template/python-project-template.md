# Python Project Template

This note covers what Python is used for in empirical economics, and a reproducible project
setup (the Python counterpart of the Stata do-file template). 

Reference throughout:
[Coding for Economists](https://aeturrell.github.io/coding-for-economists/index.html).

---

## What Python is for in empirical economics

Python is a general-purpose language, so in empirical work it covers the steps that Stata does
not do well, or does not do at all:

- **Getting data:** APIs and web scraping (`requests`, `BeautifulSoup`).
  [Getting Data](https://aeturrell.github.io/coding-for-economists/data-extraction.html)
- **Cleaning and wrangling** messy or large data with `pandas`.
  [Working with Data](https://aeturrell.github.io/coding-for-economists/data-intro.html)
- **Text as data:** regex, NLP, LLM-based labelling, turning text into regression-ready variables.
  [Text Analysis](https://aeturrell.github.io/coding-for-economists/text-intro.html)
- **Machine learning:** prediction and ML for causal inference.
  [Machine Learning](https://aeturrell.github.io/coding-for-economists/ml-intro.html)
- **Numerical and simulation work:** solving or simulating models (`numpy`, `scipy`).
- **Visualization and automation:** publication figures and reproducible pipelines.


---

## When Python, when Stata

- **Stata:** classic panel and cross-section econometrics, standard regressions, established
  estimators. Fast and convenient for these.
- **Python:** data acquisition, text, machine learning, simulation, large or messy data, and full
  automation. Use it when the task is outside Stata's core, or when a regression is just one step
  in a larger pipeline that also involves scraping, text, or ML.
- Regressions run in Python too (`statsmodels`; `pyfixest` for high-dimensional fixed effects,
  like Stata's `reghdfe`; `linearmodels` for panel and IV), but routine panel FE / IV often still
  goes to Stata.

In short: Stata is a specialized econometrics tool; Python is general-purpose glue that connects
data collection, text, ML, computation, and automation.

## Project template

Same folder logic as the Stata template (raw is read-only; code generates everything), plus one
Python-specific essential: a **virtual environment**.

```text
project/
├── data/
│   ├── raw/            # read-only source data
│   └── clean/          # generated by code
├── code/
│   ├── 01_clean.py
│   ├── 02_analysis.py
│   └── run.py          # runs the pipeline in order
├── output/
│   ├── tables/
│   └── figures/
├── requirements.txt    # pinned package versions
└── README.md
```

### Environment

**Problem.** A computer has one global Python, and Python's features come from third-party
packages (`pandas`, `numpy`, etc.) installed with `pip`. If every project installs into that one
global Python, versions collide: project A needs `pandas 1.5`, project B needs `pandas 2.0`, and
installing one breaks the other. It also becomes hard to tell what any single project actually
depends on.

**Fix.** A virtual environment is a per-project folder (usually `.venv`) holding that
project's own Python and packages, isolated from everything else. Think of it as a separate
kitchen per project: each has its own tools, so what gets installed in one never affects another.
[Virtual Code Environments](https://aeturrell.github.io/coding-for-economists/wrkflow-environments.html)

```bash
python -m venv .venv             # 1. create the environment (makes a .venv folder)
source .venv/bin/activate        # 2. enter it (Windows: .venv\Scripts\activate)
pip install pandas numpy matplotlib statsmodels   # 3. installs only into this project
pip freeze > requirements.txt    # 4. record the exact packages and versions
deactivate                       # leave the environment when done
```

After step 2 the prompt shows `(.venv)`, meaning you are inside the environment; every `pip
install` from then on affects only this project.

**Why this is reproducibility.** `requirements.txt` is a recipe: the full list of packages and
their versions. Anyone, including you on another machine, runs `pip install -r requirements.txt`
to rebuild the identical environment, so the code produces the same results. This is the Python
analog of pinning `version 18` in Stata: Stata locks the software version, while Python depends on
many external packages, so the whole package set is locked instead.

**Do not commit `.venv`.** It is large and machine-specific. Add `.venv/` to `.gitignore` and
commit only `requirements.txt`: share the recipe, not the cooked meal. This mirrors the data rule
of committing code, not raw data.

### Script header (like the do-file header)
```python
"""01_clean.py - raw to clean. Project: <name>. Author: <name>."""
from pathlib import Path
import numpy as np
import pandas as pd

np.random.seed(12345)              # reproducibility

ROOT = Path(__file__).resolve().parents[1]   # one level up from code/
RAW, CLEAN = ROOT / "data/raw", ROOT / "data/clean"

df = pd.read_csv(RAW / "dataset.csv")
# ... clean ...
df.to_parquet(CLEAN / "analysis.parquet")
```

Key reproducibility points: relative paths via `pathlib` (never hardcode absolute paths), a fixed
seed, pinned package versions, and a `run.py` that executes the scripts in order.
[Reproducible Analysis](https://aeturrell.github.io/coding-for-economists/wrkflow-rap.html)

### Core libraries

- **pandas** — data wrangling (the everyday workhorse)
- **numpy** — arrays and numerics
- **scipy** — scientific computing and optimization
- **matplotlib** — plots (also lets-plot, seaborn, plotnine)
- **statsmodels** — regression and statistics
- **pyfixest** — high-dimensional fixed effects (like `reghdfe`)
- **linearmodels** — panel data and IV / 2SLS
- **scikit-learn** — machine learning; **econml** / **doubleml** for ML + causal
- **requests** — APIs; **BeautifulSoup** — scraping
- **re** (regex), **spaCy** / **nltk**, **transformers** — text and NLP
